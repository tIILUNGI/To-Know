# Plano de Implementação - 6 Módulos do Sistema

## VISÃO GERAL

O sistema deve ter 6 módulos principais claramente separados:
1. Dashboard
2. Cadastro de Fornecedores
3. Cadastro de Clientes
4. Processos de Aprovação
5. Processos de Avaliação e Reavaliação
6. Relatórios e Histórico

---

## ANÁLISE DO ESTADO ATUAL

### ✅Já Implementado:
- **Dashboard**: 11 estatísticas + 8 filtros
- **EntityForm**: Formulário com tabs (Identificação, Localização, Responsável, Classificação, Financeiro, Compliance)
- **EntityList**: Lista de fornecedores e clientes
- **ProcessWorkflow**: Workflow de processos
- **ProcessDetail**: Detalhe do processo com critérios
- **EvaluationForm**: Formulário de avaliação
- **EvaluationList**: Lista de avaliações
- **ReportsView**: Relatórios

### ❌Problemas Identificados:
1. **Módulo 4 e 5混淆 (Misturados)**: 
   - "Processos" no menu atual combina Aprovação + Avaliação
   - Não há separação clara entre processos de APROVAÇÃO e processos de AVALIAÇÃO/REAVALIAÇÃO
   
2. **Estrutura de rotas atual**:
   - `/processes` → Workflow de processos (deveria ser APROVAÇÃO apenas)
   - `/evaluations` → Avaliações (deveria ser AVALIAÇÃO/REAVALIAÇÃO)

3. **Navegação atual**:
   - Dashboard
   - Fornecedores
   - Clientes
   - Processos (misturado)
   - Avaliações (duplicado?)
   - Relatórios

## PLANO DE IMPLEMENTAÇÃO

### Task 1: Separar Rotas e Menu

Separar claramente os 6 módulos:

**Novas rotas em App.tsx:**
```
/                       → Dashboard
/entities/suppliers    → Lista Fornecedores  
/entities/clients      → Lista Clientes
/entities/new          → Novo Fornecedor/Cliente
/entities/:id          → Editar Entidade
/processes             → Processos de APROVAÇÃO (renomear de ProcessWorkflow)
/processes/:id         → Detalhe do Processo de Aprovação
/approvals             → NOVO: Lista de aprovações
/evaluations           → Processos de AVALIAÇÃO e REAVALIAÇÃO
/evaluations/new       → Nova Avaliação/Reavaliação
/evaluations/:id       → Detalhe da Avaliação
/reports               → Relatórios
/admin                 → Configurações (critérios)
```

**Menu em Layout.tsx:**
1. Dashboard
2. Fornecedores
3. Clientes
4. Aprovações (Processos de Aprovação)
5. Avaliações (Avaliação e Reavaliação)
6. Relatórios

---

### Task 2: Criar Component de Aprovações Separado

Criar novo componente `ApprovalsList.tsx`:
- Lista de processos de aprovação
- Filtros: Estado, Tipo, Data, Responsável
- Ações: Criar, Visualizar, Aprovar/Rejeitar

Manter `ProcessWorkflow.tsx` como "Aprovações" (apenas approval workflows).

---

### Task 3: Separar Avaliações de Processos

O `/evaluations` deve ser dedicado a:
- Avaliação de Performance (Fornecedor/Cliente)
- Avaliação de Satisfação (Fornecedor/Cliente)  
- Reavaliação (com histórico comparativo)

Criar rota `/reevaluations` para listar reavaliações pendentes.

---

### Task 4: Verificar Campos Completos

**Fornecedor (7 secções):**
1. Identificação (código, nome, NIF, tipo, estado)
2. Localização (endereço, província, município, país, contactos)
3. Responsável (nome, cargo, telemóvel, email)
4. Classificação (sector, tipo fornecimento, impacto, criticidade)
5. Financeiro (condições pagamento, moeda, banco, IBAN, volume)
6. Compliance (certificados, documentos)
7. Due Diligence (PEP, sanções, riscos, classificação)

**Cliente (6 secções):**
1. Identificação
2. Localização
3. Responsável
4. Classificação (segmento, sector, importância, potencial)
5. Comercial/Financeiro
6. Compliance/Risco

---

### Task 5: Backend - Endpoints

Criar endpoints específicos:

- `GET /api/approvals` - Lista de aprovações
- `POST /api/approvals` - Criar processo de aprovação
- `GET /api/reevaluations` - Lista de reavaliações
- `POST /api/reevaluations` - Criar reavaliação
- `GET /api/evaluations/:id/compare` - Comparar com avaliação anterior

---

### Task 6: Histórico e Rastreabilidade

Melhorar o histórico:
- Cada entidade deve ter histórico completo
- Mostrar: dados alterados, utilizador, data/hora
- Documentos anexados
- Processos associados
